# 求职路线总览

这个站点的求职路线不是为了背题，而是为了把“会什么、做过什么、遇到问题怎么解决”讲清楚。面试官真正关心的通常不是你是否把某个命令背下来，而是你能否把技术点放进业务、部署、性能、故障和协作流程里。

## 目标岗位定位

当前更适合优先准备两类岗位：

| 方向 | 适合原因 | 简历表达重点 |
| --- | --- | --- |
| Java 后端开发 | 你本身喜欢 Java，后端知识能和数据库、缓存、部署串起来 | Spring Boot、接口设计、数据库优化、Redis 缓存、项目上线 |
| DevOps / 运维开发 | 就业面更宽，能利用 Linux、Docker、K8s、CI/CD 形成差异化 | 自动化部署、容器化、监控告警、故障排查、脚本能力 |
| 全栈偏工程化 | 适合中小团队岗位，要求能前后端联调和部署 | Vue、接口联调、部署发布、问题闭环 |

建议主线是：**Java 后端能力作为核心，DevOps 能力作为求职差异化**。这样既不放弃 Java，也不会把自己限制在纯 Java 岗位。

## 能力模型

一份有竞争力的技术文档和简历，需要覆盖四层能力。

### 1. 基础能力

基础能力决定你能不能把题讲明白。

- Java：集合、异常、泛型、反射、注解、IO、多线程、JVM 基础。
- 数据库：MySQL 存储引擎、索引、事务、锁、慢 SQL、执行计划。
- Linux：文件、进程、网络、磁盘、日志、权限、systemd。
- 网络：HTTP、DNS、TCP 连接、端口、反向代理。

### 2. 框架能力

框架能力决定你能不能做真实业务。

- Spring Boot：自动配置、配置文件、Starter、Bean 生命周期、拦截器、异常处理。
- MyBatis：SQL 映射、动态 SQL、分页、批量写入、N+1 查询问题。
- Redis：缓存、分布式锁、缓存穿透、击穿、雪崩、集群与持久化。
- Vue：组件、路由、状态管理、接口请求、构建部署。

### 3. 工程能力

工程能力决定你能不能把项目交付上线。

- Git 分支管理和回滚。
- Docker 镜像构建、容器运行、日志查看。
- Nginx 静态资源托管、反向代理、HTTPS。
- Jenkins / GitHub Actions / Vercel 自动化部署。
- Prometheus / Grafana / 日志平台的基础监控思路。

### 4. 表达能力

表达能力决定面试官是否相信你真的做过。

一个技术点不要只背概念，建议按这个结构回答：

1. 它解决什么问题。
2. 核心原理是什么。
3. 在项目里怎么用。
4. 出问题如何排查。
5. 有哪些生产环境注意事项。

例如 Redis 缓存不能只说“提高性能”，还要说缓存 key 如何设计、过期时间如何设置、缓存和数据库如何保持一致、热点 key 怎么处理、缓存异常时系统如何降级。

## 学习顺序

### 第一阶段：补齐基础

先把 Java、Linux、MySQL、Redis 的基础概念补齐，不需要一开始追求大而全。

推荐顺序：

1. Linux 常用命令和日志排查。
2. MySQL 存储引擎、索引、事务。
3. Java 集合、反射、异常、多线程基础。
4. Redis 基础数据类型、缓存问题、持久化。

### 第二阶段：做出项目闭环

一个能讲的项目至少要包含：

- 登录认证或权限控制。
- 核心业务接口。
- MySQL 表设计和索引。
- Redis 缓存或分布式锁。
- Docker 部署。
- Nginx 代理。
- README 或上线文档。

项目不一定复杂，但必须能讲清楚为什么这样设计、出了问题怎么处理。

### 第三阶段：准备生产问题

面试常见追问会集中在“你遇到过什么问题”。

可以准备这些场景：

- 接口响应慢，如何定位。
- 数据库慢 SQL，如何分析执行计划。
- Redis 缓存穿透、击穿、雪崩怎么处理。
- Docker 容器启动失败如何排查。
- K8s Pod 一直重启如何排查。
- GitHub Pages 和 Vercel 部署路径不一致怎么解决。

## 文档写作原则

每篇文档尽量按固定模板写：

```md
# 知识点名称

## 解决什么问题

## 核心概念

## 执行流程

## 实战配置或代码

## 生产环境注意事项

## 常见问题排查

## 面试回答
```

这样写的好处是：你不是在堆笔记，而是在沉淀可复用的工程知识。

## 页面中直接写文档的方案

纯 VitePress 本质是静态站点，浏览器页面不能直接把 Markdown 写回 GitHub 仓库。要实现在线写文档，必须增加认证和提交能力，常见方案有三种：

| 方案 | 原理 | 优点 | 风险 |
| --- | --- | --- | --- |
| GitHub 网页编辑 | 直接在 GitHub 页面编辑 Markdown | 不用开发，安全稳定 | 体验一般，目录组织仍靠自己 |
| Decap CMS / TinaCMS | 在站点加管理后台，通过 GitHub OAuth 提交文件 | 可以像后台一样写文章 | 需要配置 OAuth，初期复杂 |
| Vercel Serverless + GitHub API | 自己做编辑页面，服务端创建 commit 或 PR | 最自由，可定制 | 需要写鉴权，不能把 token 暴露在前端 |

当前建议先不急着做 CMS。先把目录、导航、文档模板稳定下来；等内容规模上来，再接一个安全的在线编辑方案。
