# Java 后端面试主线

Java 后端面试不要只按“八股文”准备，更应该按一个后端请求从进入系统到返回结果的链路来准备：请求进入 Nginx，转发到 Spring Boot，业务层处理，访问数据库和缓存，最后通过日志、监控和部署体系保证系统可维护。

## 一条请求的完整链路

```text
浏览器 / App
  -> Nginx
  -> Spring Boot Controller
  -> 参数校验 / 登录鉴权 / 拦截器
  -> Service 业务逻辑
  -> Redis 缓存
  -> MySQL 数据库
  -> 日志记录 / 异常处理
  -> JSON 响应
```

面试时，很多问题都可以放回这条链路中理解。

## Java 基础

### 集合

重点准备：

- `ArrayList` 和 `LinkedList` 的区别。
- `HashMap` 的底层结构、扩容机制、哈希冲突。
- `ConcurrentHashMap` 为什么适合并发场景。
- `HashSet` 为什么不能存重复元素。

回答集合问题时，不要只说 API，要补充使用场景。

例如：

> `ArrayList` 底层是数组，随机访问快，适合读多、按下标访问多的场景；插入和删除如果发生在中间位置，需要移动元素。`LinkedList` 底层是链表，理论上插入删除更灵活，但实际业务里由于缓存局部性和额外对象开销，很多时候 `ArrayList` 仍然更常用。

### 异常

重点准备：

- `Exception` 和 `RuntimeException` 的区别。
- 为什么业务异常通常继承运行时异常。
- 全局异常处理如何返回统一格式。
- 异常日志应该记录什么。

生产实践里，异常处理不只是 `try-catch`，还包括：

- 给用户返回清晰但不泄露内部细节的信息。
- 给日志记录足够定位问题的上下文。
- 对可预期业务错误使用业务异常。
- 对系统错误保留堆栈并告警。

### 反射与注解

反射要和 Spring 联系起来讲。

Spring Boot 中常见的反射和注解场景：

- 扫描 `@Component`、`@Service`、`@Controller` 创建 Bean。
- 读取 `@Autowired`、`@Value` 完成依赖注入。
- 读取 `@RequestMapping` 建立 URL 到方法的映射。
- MyBatis 通过接口和代理对象执行 SQL。

面试回答可以这样组织：

> 反射允许程序在运行期获取类、方法、字段和注解信息，并动态创建对象或调用方法。框架用反射把“写死的代码调用”变成“根据配置和注解决定行为”，从而实现 IoC、AOP、ORM 等能力。

## Spring Boot

### 自动配置

Spring Boot 自动配置的核心思路是：根据 classpath 中是否存在某些类、配置文件中是否开启某些属性，自动创建默认 Bean。

要能说清楚：

- `@SpringBootApplication` 包含 `@SpringBootConfiguration`、`@EnableAutoConfiguration`、`@ComponentScan`。
- 自动配置不是魔法，本质是条件装配。
- 业务中可以通过自定义 Bean 覆盖默认配置。

### 分层结构

推荐项目分层：

```text
controller  接收请求、参数校验、返回响应
service     编排业务流程、处理事务
mapper      数据库访问
domain      实体、枚举、值对象
config      配置类
common      通用返回、异常、工具类
```

面试官问“为什么要分层”，可以回答：

- 降低耦合。
- 方便测试。
- 让业务逻辑集中在 Service。
- Controller 不直接写 SQL，避免接口层变得混乱。

### 事务

重点准备：

- `@Transactional` 作用在哪一层。
- 事务什么时候会失效。
- 默认只回滚运行时异常。
- 同类方法内部调用可能导致事务不生效。
- 长事务会占用连接和锁资源。

生产环境里，事务边界要尽量小。不要在事务里做远程调用、文件上传、复杂循环、长时间等待。

## MySQL

### 必备知识点

- InnoDB 和 MyISAM 区别。
- B+Tree 索引结构。
- 聚簇索引和二级索引。
- 最左前缀原则。
- 覆盖索引。
- 事务隔离级别。
- MVCC。
- 行锁、间隙锁、临键锁。
- 慢 SQL 和 `EXPLAIN`。

### SQL 优化回答模板

遇到 SQL 慢，可以按这个流程回答：

1. 先确认慢 SQL 是否稳定复现。
2. 查看数据量、返回行数、过滤条件。
3. 使用 `EXPLAIN` 分析访问类型、使用索引、扫描行数。
4. 判断索引是否合理，是否满足最左前缀。
5. 避免函数、隐式类型转换、前置 `%like` 导致索引失效。
6. 必要时改写 SQL、增加联合索引、拆分大查询。
7. 上线后观察慢日志和接口耗时。

## Redis

### 缓存问题

常见三件套：

- 缓存穿透：查询不存在的数据，打到数据库。解决：缓存空值、布隆过滤器、参数校验。
- 缓存击穿：热点 key 失效，大量请求打到数据库。解决：互斥锁、逻辑过期、热点 key 预热。
- 缓存雪崩：大量 key 同时失效或 Redis 故障。解决：过期时间随机化、限流降级、多级缓存、高可用。

### 数据一致性

常用策略是先更新数据库，再删除缓存。原因是缓存删除失败可以通过重试、消息队列、定时校验补偿；如果先删缓存再更新数据库，高并发下更容易读到旧数据并写回缓存。

面试里不要说“保证强一致”，多数缓存场景追求的是最终一致。

## 项目表达

一个 Java 项目至少准备三个故事：

### 1. 业务故事

项目解决什么问题，核心模块是什么，用户如何使用。

### 2. 技术故事

为什么选择 Spring Boot、MySQL、Redis、Docker，数据表怎么设计，接口怎么拆。

### 3. 问题故事

遇到过什么问题，如何定位，如何修复，最后如何防止再次发生。

如果暂时没有真实线上事故，可以从本地部署、接口慢、SQL 不走索引、容器启动失败这些小问题开始复盘。真实感来自过程，不来自夸大规模。
